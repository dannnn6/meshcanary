# Mesh Canary

[Страница проекта](https://YOUR_USERNAME.github.io/meshcanary/) · MIT license

Распределённая, не имеющая единого центра сеть для мониторинга доступности
сайтов. Идея: вместо одного сервера, который проверяет «доступен ли сайт X»
(и который можно заблокировать, перегрузить или взломать), десятки
независимых добровольных узлов проверяют сайты каждый со своей точки
интернета и подписывают результат своим ключом. Узлы обмениваются
подписанными отчётами друг с другом — и у каждого участника в итоге
оказывается своя, независимо проверяемая копия общей картины. Никто не
может её ни стереть, ни подделать задним числом, потому что для этого
пришлось бы подделать чужую цифровую подпись.

Это близко по духу к проекту OONI (Open Observatory of Network
Interference), но без центрального backend'а — данные живут в самой сети,
а не в одной базе данных.

## Как это устроено

**Личность узла = его публичный ключ.** Никакого реестра пользователей —
при первом запуске узел сам генерирует пару ключей Ed25519
(`node.key` — приватный, остаётся на диске; публичный ключ в hex-виде
*и есть* идентификатор узла). Это та же модель, что у Tor и SSH.

**Отчёт** — это маленький подписанный документ:
```
{
  "node_id":    "<hex публичного ключа>",
  "target":     "github.com",
  "timestamp":  "2026-06-21T19:00:27+00:00",
  "status":     "OK" | "DNS_FAIL" | "TCP_TIMEOUT" | "TCP_REFUSED" | "TLS_FAIL",
  "latency_ms": 142,
  "sig":        "<подпись Ed25519 над каноническим JSON остальных полей>"
}
```
Подпись можно проверить, зная только сам отчёт — не нужно доверять узлу,
который его переслал.

**Проверка цели** (`check_target`): DNS-резолв → TCP-подключение на 443
порт → TLS handshake. Если что-то из этого падает — статус показывает,
на каком именно этапе (это и есть полезный сигнал: `DNS_FAIL` обычно
значит DNS-блокировку, `TCP_TIMEOUT`/`TCP_REFUSED` — блокировку на
уровне провайдера, `TLS_FAIL` — перехват/подмену сертификата).

**Gossip-протокол** — простой push-pull: каждые `--gossip-interval`
секунд узел подключается к нескольким известным ему пирам, отправляет
свои последние отчёты, получает в ответ их отчёты, проверяет подписи и
сохраняет новые. Со временем у всех узлов сходится одна и та же картина,
даже если они никогда не подключаются ко всем остальным напрямую —
отчёты распространяются транзитивно, как сплетни.

**Автообнаружение пиров (peer exchange).** Вместе с отчётами узлы
обмениваются ещё и списком известных им адресов других узлов — точно
так же, как PEX в BitTorrent или addr-gossip в Bitcoin. Это значит, что
новому узлу достаточно знать **один** bootstrap-адрес: дальше сеть сама
расскажет ему про остальных. Полноценный Kademlia DHT для этого не
нужен — он избыточен для сети из десятков-сотен волонтёрских узлов и
тянет за собой лишнюю сложность; gossip-обмен даёт тот же практический
результат и его легко понять, читая код.

**Постоянное хранилище.** Все отчёты и список известных пиров лежат в
SQLite (`--db meshcanary.db`). Перезапуск узла не стирает историю —
файл просто открывается заново.

**Веб-дашборд.** Если указать `--web-port`, узел поднимает встроенный
HTTP-сервер (без внешних зависимостей) с живой страницей на
`http://localhost:<port>/` и JSON-эндпоинтом на `/api/status` — удобно
смотреть глазами, а не читать консольные логи.

## Запуск локальной демонстрации (3 узла на одной машине)

```bash
pip install cryptography --break-system-packages

# Каждую команду — в своём терминале.
# Обрати внимание: B и C в demo/peers_b.json и demo/peers_c.json знают
# только про A — друг про друга они узнают сами, через peer exchange.
python3 node.py --port 9001 --id-file demo/a.key --db demo/a.db \
  --peers demo/peers_a.json --advertise-host 127.0.0.1 --web-port 9101

python3 node.py --port 9002 --id-file demo/b.key --db demo/b.db \
  --peers demo/peers_b.json --advertise-host 127.0.0.1 --web-port 9102

python3 node.py --port 9003 --id-file demo/c.key --db demo/c.db \
  --peers demo/peers_c.json --advertise-host 127.0.0.1 --web-port 9103
```

Открой `http://localhost:9102/` в браузере — увидишь живую сводку.
В логе узла B (который изначально знал только про A) через пару циклов
появятся строчки `gossip: получено N новых отчётов от 127.0.0.1:9003` —
это и есть доказательство, что он сам нашёл узел C.

## Запуск на разных машинах (по-настоящему распределённо)

1. На каждой машине: `python3 node.py --port 9001 --advertise-host <свой
   публичный IP или домен> --web-port 8080` (порт `--port` должен быть
   проброшен наружу — открыт в файрволе / на роутере; `--advertise-host`
   обязателен, иначе остальные не смогут включить тебя в свой peer
   exchange).
2. На первом узле в `peers.json` впиши адреса пары известных тебе
   участников сети — этого достаточно, остальных узел найдёт сам:
   `{"peers": [{"host": "1.2.3.4", "port": 9001}]}`.
3. Все узлы должны указывать на один и тот же `targets.json` (список
   сайтов для проверки) — либо синхронизируй его вручную, либо просто
   договоритесь о списке.

## Автозапуск и фоновый режим (systemd)

`install.sh` автоматически ставит недостающие зависимости, регистрирует
systemd-сервис и запускает ноду в фоне. Больше ничего делать не нужно.

```bash
journalctl -u meshcanary -f          # логи в реальном времени
sudo systemctl restart meshcanary    # перезапустить (запускай после update.sh)
sudo systemctl stop meshcanary       # остановить
```

## Обновление

```bash
bash update.sh
```
Скрипт обращается к `git remote` (твоему GitHub-репозиторию), сравнивает
твою версию с последней доступной, и если есть разница — показывает:
номер текущей и новой версии, список новых коммитов и полный `git diff`
по `node.py`, `common/` и `install.sh`. Применяет изменения только после
твоего подтверждения (`y`), и только fast-forward'ом — если у тебя есть
несохранённые локальные правки, скрипт откажется обновляться и попросит
сделать `git stash`, а не молча их затереть.

Версия отслеживается в файле `VERSION` (semver: `MAJOR.MINOR.PATCH`).
Если вносишь изменения и хочешь, чтобы `update.sh` показывал их твоим
пользователям как новую версию — не забудь бампнуть `VERSION` в том же
коммите.

`targets.json` и `peers.json` (личные настройки) специально не лежат в
git — это копии `targets.json.example` / `peers.json.example`,
создаваемые `install.sh`. Поэтому `update.sh` никогда не будет
конфликтовать с твоими личными списками целей и пиров.

## Ограничения текущей версии и куда расти дальше

- **Peer exchange, а не полноценный DHT.** Подходит для сети из
  десятков-сотен узлов. Если сеть когда-нибудь дорастёт до тысяч
  участников, имеет смысл перейти на настоящий Kademlia (`libp2p` /
  `kademlia`) для логарифмической маршрутизации — но это уже совсем
  другой масштаб задачи.
- **Нет защиты от Sybil-атаки.** Кто угодно может поднять много
  фальшивых узлов и «забить» сеть отчётами вида «всё доступно».
  Подписи гарантируют только то, что отчёт *действительно* от этого
  node_id, но не мешают создать сколько угодно node_id. Для серьёзного
  использования нужна репутационная система или web-of-trust между
  операторами узлов.
- **Список целей — не децентрализован.** Сейчас `targets.json` —
  локальный файл. Можно сделать его самим расходящимся (gossip) списком,
  чтобы добавлять новые цели для мониторинга могла сама сеть, с подписями
  и голосованием.
- **Гео-метка не собирается автоматически** — намеренно: автоматическое
  определение местоположения участников по IP без их согласия — плохая
  практика для проекта, который сам про privacy. Если захочешь
  географическую карту блокировок, добавь в отчёт необязательное поле
  `region`, которое оператор узла указывает сам.
- **Дашборд без TLS/аутентификации.** Это локальный инструмент для
  оператора узла, не предназначенный для выкладывания в открытый
  интернет как есть — если захочешь публичную страницу статуса,
  поставь перед ним nginx с TLS и базовой авторизацией.

## Этичное использование

Список целей должен состоять из публичных сервисов (новостные сайты,
соцсети, мессенджеры и т.д.), доступность которых интересна многим — а
не из чьего-то личного сервера. Это инструмент для наблюдения за общей
доступностью интернета, а не для слежки за конкретным человеком или
организацией.

## Структура репозитория

```
meshcanary/
├── node.py              узел: пробы, gossip-сервер/клиент, дашборд
├── install.sh            установка (venv + зависимости + run-node.sh)
├── VERSION                текущая версия (semver)
├── targets.json.example   шаблон списка целей для проверки
├── peers.json.example     шаблон bootstrap-списка пиров
├── common/
│   ├── crypto.py          Ed25519: identity, подпись, проверка
│   ├── models.py          формат отчёта, каноническая сериализация
│   ├── protocol.py        сетевой протокол (фреймирование JSON)
│   ├── storage.py         персистентное хранилище (SQLite) + очистка
│   └── dashboard.py       встроенный веб-дашборд
├── demo/                  готовые конфиги для локальной демонстрации 3 узлов
├── docs/
│   └── index.html         страница проекта (GitHub Pages)
└── .github/workflows/
    └── ci.yml             проверка синтаксиса и smoke-test при пуше
```
